## Hook
The riskiest AI thing in your company next year won’t be the model.
It’ll be the agents you quietly wired into production systems.

## Core Idea
- Most teams are still thinking about AI security as "prompt + data" and ignoring the fact that agents are effectively unreviewed junior engineers with API keys.
- In real projects, the worst risks come from how agents are orchestrated: tool permissions, autonomy, and lack of observability — not from the base model itself.
- There are four recurring security mistakes in agent projects: insecure tools, over-trusted autonomy, no audit trail, and shadow agents built outside governance.
- Treating agents as "smart endpoints" instead of "unreliable interns" leads to non-obvious failure modes: lateral movement via tools, business logic abuse, and untraceable incidents.
- You can reduce 80% of the risk with a few simple patterns: least-privilege tools, constrained interfaces, forced logging, and an explicit inventory of agents and their blast radius.

---
Slide 1 — The uncomfortable truth
Title: Your most dangerous "employee" is a JSON config file

Body:
When you add tools to an agent, you’re not just improving UX.
You’re giving a stochastic system the ability to:
– change data
– trigger workflows
– touch real customers

Most teams lock down the model and the data.
Almost nobody locks down the actions.

This is where the real security story starts.

---
Slide 2 — Mistake #1
Title: Treating tools as toys, not capabilities
Subtitle: "It’s just a POC"… wired into production

What goes wrong:
– One shared service account for all tools
– Agents can call any tool from any context
– No environment separation (dev/stage/prod)

Real pattern I keep seeing:
– A "run_sql" tool with full prod access
– No row-level security
– No query whitelisting or approval step

What to do instead:
– Design tools as capabilities with a blast radius
– One agent = one role = scoped permissions
– Separate credentials per environment
– Default to read-only tools; add write only when a human approves

Non-obvious pattern:
The biggest risk window is the "experimental" phase — that’s when people bypass all the normal security processes.

---
Slide 3 — Mistake #2
Title: Over-trusting agent autonomy
Subtitle: You gave it root access and prayed the prompt was enough

What goes wrong:
– Assuming "the model won’t do that"
– Free-form natural language → powerful tools with no guardrails
– No business-logic checks on tool arguments

Example pattern:
– Agent can:
  – create tickets
  – send emails
  – adjust discounts
– The *only* control is natural language instructions in the system prompt.

This isn’t a guardrail. It’s a wish.

What to do instead:
– Wrap tools with explicit schemas and hard constraints
– Validate all tool inputs against business rules
– Require human-in-the-loop for high-impact actions (money, access, data deletion)
– Treat the agent as an unreliable intern whose work must be checked

Non-obvious pattern:
Most "prompt injection" risks become much worse when the agent has tools that modify state.

---
Slide 4 — Mistake #3
Title: No audit trail for what the agent actually did
Subtitle: You can’t secure what you can’t reconstruct

What goes wrong:
– Only log final user-visible output
– No record of intermediate prompts, tool calls, or responses
– No linkage between a user, a request, and the chain of actions

Real impact:
– You can’t debug bad behavior
– You can’t do forensics after an incident
– You can’t improve the agent systematically

What to do instead:
Log at the orchestration level:
– Full prompt + tool call history (with redaction where needed)
– Who/what initiated each request
– Which credentials / environment were used
– Outcome of each tool call (including failures)

Non-obvious pattern:
Most orgs have better logging for a cron job that runs once a day than for an agent that can touch thousands of records an hour.

---
Slide 5 — Mistake #4
Title: Shadow agents: the new shadow IT
Subtitle: "It’s just a small automation" (with production access)

What goes wrong:
– Teams spin up agents in:
  – notebooks
  – Zapier / n8n
  – internal scripts
– They reuse prod API keys they already have
– Security and platform teams never hear about it

Pattern I keep seeing:
– First: one-off "helper" agent that writes Slack updates
– Then: it gets tool access to ticketing / CRM / billing
– Six months later: nobody knows what it can actually do

What to do instead:
– Maintain an inventory of agents:
  – where they run
  – which tools they have
  – who owns them
– Require a minimal threat model for any agent with write access
– Make "agent reviews" part of your existing change/governance process

Non-obvious pattern:
Most agent-related incidents in the next 12–18 months will come from these shadow systems, not the flagship AI product.

---
Slide 6 — Putting it together
Title: A simple mental model: agents as untrusted code + powerful tools

If you only remember one thing, make it this:

An agent is not a UI feature.
It’s untrusted code that:
– writes its own plans
– chooses its own tools
– executes real actions

Minimum viable security for agents:
1) Least-privilege tools with explicit blast radius
2) Hard constraints + validation on tool inputs
3) Orchestration-level logging and traceability
4) An actual inventory of agents and who owns them

If you’re already building with agents:
– Which of these 4 mistakes are you making today?
– What would break first if the agent stopped following the prompt?

That’s where your security work should start.
