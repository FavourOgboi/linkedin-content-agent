## Hook
Are AI Agents Your Next Security Nightmare? (What Everyone’s Missing About ‘Autonomy’)

## Core Idea
- Most teams still treat agents as fancy chatbots, not security principals with real permissions.
- The main risk isn’t the base model – it’s the action space you expose and how you bind tools, data, and credentials.
- There are repeatable anti-patterns: over-trusting tool output, unconstrained action spaces, brittle guardrails, and no separation between ‘thinking’ and ‘doing’.
- You can borrow proven patterns from security engineering (least privilege, blast radius, capability tokens, auditability) and apply them to LLM agents before you ship them into production.

## Draft Post
[Slide 1]
Title: AI Agents Aren’t Smart Assistants. They’re New Users With Root Access.

Everyone is stress-testing models.
Almost no one is stress-testing what their agents are allowed to *do*.

If you give an LLM the ability to:
- call APIs
- write to your DB
- send emails / Slack
- touch cloud resources

…you didn’t just add “autonomy”.
You created a new, barely-understood security principal.

That’s where the real risk lives.

This carousel is about the mistakes I keep seeing in early agent systems – and the non-obvious patterns that actually reduce blast radius.

---
[Slide 2]
Title: Mistake #1 — Treating “Autonomy” as a UX Feature, Not a Security Boundary

How most teams reason:
“Let’s make the agent as hands-off as possible. Fewer human approvals = better UX.”

What they miss:
Every new tool you attach is a *new capability*.
Capabilities compose.

Read-only DB + file write + Slack send
= data exfiltration via Slack

The non-obvious pattern:
Design agents like you design privileged service accounts:
- Explicit threat model
- Explicit capabilities
- Explicit monitoring

If you wouldn’t give a junior engineer unrestricted prod access on day one, don’t give it to a v0 agent.

---
[Slide 3]
Title: Mistake #2 — Confusing Intent Alignment With Action Safety

“I added a system prompt with rules. The agent knows not to do bad things.”

This is model-centric thinking.

Reality:
- Prompts are *suggestions*, not enforcement.
- Prompt injection works alarmingly well.
- Tool call patterns get weird under distributional shift.

Non-obvious pattern:
Treat the LLM as *untrusted planning logic*.

The real guardrails live outside the model:
- Hard constraints on which tools can be called in which contexts
- Schema-level validation on every action
- Allowlists, not rely-on-the-prompt-lists

If your only safety layer is the system prompt, you don’t have a safety layer.

---
[Slide 4]
Title: Mistake #3 — Insecure Action Spaces (The Glue Is the Vulnerability)

Everyone audits:
- The model (evals, jailbreaks)
- The tools (auth, rate limits)

Almost no one audits:
- The *binding* between LLM and tools

Common anti-patterns I keep seeing:
- Natural-language tool descriptions that are overly broad
- Single catch-all “execute_code” tool with filesystem + network access
- DB tools that allow arbitrary SQL with prod credentials

Non-obvious pattern:
Your action space should be:
- Narrow: small, focused tools
- Typed: strict schemas, enums, and limits
- Contextual: tools only available in specific workflows / states

The more generic your tools, the more creative your failure modes.

---
[Slide 5]
Title: Mistake #4 — Over-Trusting Tool Output (Silent Compromise)

Most agent stacks do:
1) Model decides to call tool
2) Tool runs
3) Model trusts the tool output as ground truth

Two hidden problems:
- Tools can be compromised (e.g., prompt-injected retrieval, poisoned data)
- Tools can be misused (wrong params, wrong context)

Non-obvious pattern:
Sanity-check tools *and* their use:
- Cross-validate critical actions with a second tool or heuristic check
- Add cheap invariants (e.g., “rows changed < 1000” for DB writes)
- Log and diff state transitions: what exactly changed after each tool call?

Most scary incidents won’t look like a jailbreak.
They’ll look like a quiet series of “valid” tool calls.

---
[Slide 6]
Title: Mistake #5 — No Concept of Blast Radius for Agents

Teams often ask: “Can the agent do X?”
They rarely ask: “If this agent goes rogue, what’s the *worst* it could do?”

Patterns I’m seeing:
- Single agent identity with:
  - full prod DB access
  - ability to send external emails
  - access to internal documentation
- Same keys used in dev and prod

Non-obvious pattern:
Design agents with blast radius in mind:
- Environment separation (dev/stage/prod agents, different keys)
- Fine-grained scopes per workflow (read vs write, table-level, resource-level)
- Time-bounded capabilities (short-lived tokens, expiring sessions)

You’ll never get risk to zero.
You *can* make the worst case survivable.

---
[Slide 7]
Title: Mistake #6 — Mixing “Thinking” and “Doing” in One Agent

Typical design:
One agent that:
- plans
- decides
- executes

This is convenient.
It’s also hard to secure.

Why it’s a problem:
- No natural choke point to insert approvals
- No separation between reasoning errors and execution errors
- Hard to debug: you only see “the agent did something weird”

Non-obvious pattern:
Split your system:
- Planner agent: proposes a plan (natural language)
- Executor: a thin, highly-constrained layer that turns approved plans into actions

You’d never let a junior analyst both architect a migration and run it directly in prod.
Treat agents the same way.

---
[Slide 8]
Title: Mistake #7 — Assuming Logs = Security

“Yes, but we log everything the agent does.”

Logging is necessary.
It’s not sufficient.

Common issues:
- Logs are unstructured, impossible to query
- No defined alerts for dangerous patterns
- No replay/repro mechanism

Non-obvious pattern:
Design logs for:
- Forensics: full chain of thought → tools → state changes
- Diffing: before/after snapshots for critical resources
- Simulation: the ability to replay an interaction in a sandbox

If you only look at logs *after* an incident, you’re already late.

---
[Slide 9]
Title: What To Do Before You Ship Your First Agent to Production

Here’s a minimal checklist I now treat as non-negotiable:

1) Threat model the *capabilities*, not the model.
2) Enumerate tools and define:
   - exact scopes
   - rate limits
   - environments
3) Separate planning from execution.
4) Add invariants and sanity checks around tool use.
5) Implement least privilege and blast-radius limits.
6) Test agents in a sandbox with:
   - prompt injection
   - adversarial tool outputs

Most orgs don’t need fewer agents.
They need agents designed as security-aware systems, not clever demos.

If you’re building agents today, how are you constraining what they’re allowed to *do*?
